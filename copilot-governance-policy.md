## 第1章 総則

### 第1条（目的）

本規程は、当社における GitHub Copilot（以下「Copilot」）の利用に関して、JDLA「生成AIの利用ガイドライン」およびAIガバナンス・エコシステムの考え方に基づき、以下を目的として定める。

1. Copilot の安全かつ効果的な利用を推進し、開発生産性の向上を図ること
2. 機密情報・個人情報の漏洩リスクを防止すること
3. 知的財産権の侵害を防止すること
4. AI生成コードの品質・セキュリティを確保すること
5. 法令・社内規程・顧客契約への適合を担保すること

### 第2条（適用範囲）

1. **対象者:** 当社の役員および従業員（業務委託先・派遣社員を含む）で、Copilot を業務に使用する全ての者
2. **対象ツール:** GitHub Copilot Business / Enterprise（個人向けプラン Individual / Pro / Pro+ は業務利用を禁止する）
3. **対象業務:** 当社の業務に関連するソフトウェア開発の全工程（設計補助、コーディング、テスト生成、ドキュメント生成、コードレビュー補助等）

### 第3条（定義）

本規程において使用する用語の定義は以下のとおりとする。

| 用語 | 定義 |
|---|---|
| AI生成コード | Copilot が提案・生成したコード、コメント、ドキュメント等の出力物 |
| プロンプト | Copilot に対する指示・質問・コンテキストとして入力される情報 |
| 機密情報 | 当社の営業秘密、顧客情報、未公開の技術情報その他秘密保持の対象となる情報 |
| コンテンツフィルター | 公開リポジトリとの一致コードや不適切なコードの提案をブロックする Copilot の機能 |
| Content Exclusion | 特定のファイル・リポジトリを Copilot の参照対象から除外する機能 |
| パブリックコードフィルター | 公開コードと一致する提案を検出・ブロックするフィルター機能 |

## 第2章 ガバナンス体制

### 第4条（責任体制）

Copilot の利用に関するガバナンス体制は以下のとおりとする。

| 役割 | 責任 | 任命部署 |
|---|---|---|
| **AI-CoE（AI推進委員会）** | Copilot 利用方針の策定・承認、効果測定、規程改廃の審議 | 経営企画部 |
| **AI責任者** | ガバナンス全体の統括、経営層への報告 | AI-CoE委員長 |
| **情報セキュリティ管理者** | セキュリティポリシーの策定・監査、インシデント対応の統括 | 情報システム部 |
| **GitHub Organization 管理者** | ライセンス管理、ポリシー設定、利用ログの監視 | 情報システム部 |
| **プロジェクト責任者** | 担当プロジェクトにおける Copilot 利用可否の判断、顧客との合意確認 | 各事業部 |
| **利用者（開発者）** | 本規程の遵守、AI生成コードのレビュー・品質確保 | 各部署 |

### 第5条（意思決定プロセス）

1. Copilot の利用方針に関する重要事項は、AI-CoE の審議を経て決定する
2. 新規プロジェクトへの Copilot 適用可否は、プロジェクト責任者が第4章の基準に基づき判断する
3. 本規程の例外適用は、AI責任者の承認を要する

### 第6条（外部ステークホルダーとの連携）

JDLA AIガバナンス・エコシステムの考え方に基づき、以下の外部連携を行う。

1. GitHub 社のサービス利用規約・プライバシーポリシーの変更を継続的に監視する
2. JDLA・経産省・総務省等が公表するガイドライン・法規制の動向を定期的に確認する
3. 業界団体・他社の事例を参考に、運用ルールの改善を図る

## 第3章 利用が禁止される用途

### 第7条（禁止用途）

以下の用途での Copilot の利用を禁止する。

1. **法令違反を目的とするコード生成:** マルウェア、不正アクセスツール、違法行為を支援するコードの生成
2. **個人の権利を侵害する目的での利用:** 特定個人の監視、プロファイリング、差別を目的とするコード生成
3. **顧客契約で AI 利用が禁止されているプロジェクトでの利用:** 顧客との契約上、AI ツールの使用が明示的に禁止されている業務
4. **機密性の極めて高いシステムでの利用:** 国家安全保障に関わるシステム、暗号鍵管理システム等
5. **安全性が重視されるシステムの自律的コード生成:** 医療機器制御、航空管制等の人命に関わるシステムにおいて、人間のレビューなしにコードを適用すること
6. **Individual / Pro / Pro+ プランでの業務利用:** Business / Enterprise 以外のプランでは、コードがAIモデルの学習データに使用される可能性があるため

### 第8条（禁止用途の判断基準）

前条の禁止用途に該当するか判断が困難な場合は、プロジェクト責任者が情報セキュリティ管理者と協議のうえ判断する。判断の記録は文書として保存する。

## 第4章 利用が許可される範囲

### 第9条（利用可能なライセンス）

| プラン | 業務利用 | 理由 |
|---|---|---|
| GitHub Copilot Business | **許可** | コードが学習データに使用されない契約保証あり |
| GitHub Copilot Enterprise | **許可** | 上記に加え、組織カスタマイズ・監査ログ拡張あり |
| GitHub Copilot Individual / Pro / Pro+ | **禁止** | 学習データ使用のオプトアウト設定が必要であり、組織管理機能がない |

### 第10条（利用可能な機能）

| 機能 | 利用可否 | 条件 |
|---|---|---|
| IDE コード補完（インライン提案） | **許可** | 第6章のレビュー規則に従うこと |
| Copilot Chat（IDE内） | **許可** | 第5章の入力制限に従うこと |
| Copilot Chat（github.com） | **許可** | 同上 |
| Copilot CLI | **許可** | 同上 |
| Copilot Agent（自律実行モード） | **条件付き許可** | プロジェクト責任者の承認を要する。実行結果は必ず人間がレビューする |
| MCP（Model Context Protocol）連携 | **条件付き許可** | 接続先サーバーの安全性を情報セキュリティ管理者が事前承認する |
| Copilot Extensions（第三者拡張） | **条件付き許可** | 情報セキュリティ管理者が事前に安全性を評価・承認したもののみ |

### 第11条（プロジェクト単位の利用可否判断）

プロジェクト責任者は、以下のチェックリストに基づき利用可否を判断する。

| # | 確認項目 | 判定 |
|---|---|---|
| 1 | 顧客契約にAIツール利用の禁止条項がないか | ☐ |
| 2 | 取り扱うデータに特別な機密区分（極秘等）がないか | ☐ |
| 3 | 業界固有の規制（金融、医療等）でAI利用が制限されていないか | ☐ |
| 4 | Content Exclusion の設定が必要なリポジトリを特定したか | ☐ |
| 5 | プロジェクトメンバーが本規程に基づく研修を修了しているか | ☐ |

上記全項目を確認し、問題がない場合に Copilot の利用を許可する。判断結果はプロジェクト管理台帳に記録する。

## 第5章 データ入力に関する規則

### 第12条（入力が禁止される情報）

Copilot への入力（プロンプト、コメント、ソースコード中の記述を含む）に以下の情報を含めてはならない。

| # | 禁止情報の種類 | 具体例 |
|---|---|---|
| 1 | **認証情報・シークレット** | API キー、パスワード、トークン、秘密鍵、接続文字列 |
| 2 | **個人情報** | 氏名、住所、電話番号、メールアドレス、マイナンバー、生年月日 |
| 3 | **顧客データ** | 顧客の業務データ、契約情報、財務データ |
| 4 | **秘密保持義務を負う情報** | NDA 対象の技術情報、他社から開示された機密情報 |
| 5 | **社内の未公開情報** | 未発表の製品仕様、M&A情報、人事情報 |
| 6 | **要配慮個人情報** | 病歴、犯罪歴、信条、人種・民族に関する情報 |

### 第13条（入力情報の安全管理措置）

1. **シークレット管理:** 認証情報は環境変数またはシークレット管理ツール（GitHub Secrets, AWS Secrets Manager 等）で管理し、ソースコード中に直接記述しない
2. **.gitignore の設定:** 機密情報を含むファイル（.env, credentials.json 等）は .gitignore に追加する
3. **Content Exclusion の活用:** 機密性の高いリポジトリ・ディレクトリは、GitHub Organization の設定で Copilot の参照対象から除外する
4. **シークレットスキャン:** GitHub Secret Scanning を有効化し、認証情報のコミットを自動検出・ブロックする
5. **プレコミットフック:** 機密情報パターンを検出するプレコミットフックを導入する

### 第14条（第三者の著作物の入力）

1. 第三者が著作権を有するソースコードを Copilot に入力する場合は、当該著作物のライセンス条件に従う
2. ライセンスが不明なコードを Copilot に入力しない
3. 顧客から預かったソースコードの入力は、顧客の書面による許諾がある場合に限る

## 第6章 生成物（出力コード）の利用に関する規則

### 第15条（AI生成コードのレビュー義務）

1. AI生成コードは、**必ず人間がレビューしてから** 採用すること。無レビューでの本番コードへの適用を禁止する
2. レビューでは以下の観点を確認する:
   - コードの正確性・意図との整合性
   - セキュリティ脆弱性の有無
   - 社内コーディング規約への準拠
   - 既存コードベースとの整合性
   - ライセンス上の問題の有無
3. **Copilot Code Review（自動レビュー機能）は一次スキャンとして活用し、人間による最終承認を省略しないこと。** AI による自動レビューは、人間レビュアーの判断を補助するものであり、代替するものではない
4. AI生成コードを含む Pull Request では、通常のコードと同等以上のレビュー基準を適用する。特に Copilot Agent / Coding Agent が自動生成した PR については、人間による2名以上のレビュー承認を必須とする
5. レビュー記録（承認者・指摘事項・対応内容）は Pull Request 上に残し、監査証跡として保管すること

### 第16条（AI生成コードの品質基準）

AI生成コードは、人間が記述したコードと同等の品質基準を満たす必要がある。

| 基準 | 要件 |
|---|---|
| テスト | AI生成コードに対しても通常と同等のテストを実施すること |
| 静的解析 | 静的解析ツール（SAST）による自動スキャンを通過すること |
| コードレビュー | Pull Request での他者レビューを経ること |
| CI/CD | 既存の CI/CD パイプラインでのビルド・テストに合格すること |

### 第17条（ハルシネーション対策）

Copilot は存在しない API、関数、ライブラリ、パッケージを提案することがある（ハルシネーション）。以下の対策を講じること。

1. AI が提案した外部ライブラリ・パッケージの実在性を確認する
2. CI/CD パイプラインでのビルド・テスト実行により、存在しない依存関係を検出する
3. 提案されたAPIの公式ドキュメントとの照合を行う

### 第18条（AI生成コードの虚偽・不正確性への対処）

1. Copilot の生成物には誤りが含まれる可能性があることを常に認識する
2. ビジネスロジックの正確性は開発者自身が責任を持って検証する
3. AI生成コードに起因する不具合・障害が発生した場合も、通常の不具合と同様のインシデント対応フローに従う

## 第7章 知的財産・ライセンス管理

### 第19条（著作権リスクへの対応）

1. **パブリックコードフィルターの有効化:** GitHub Organization の設定で、パブリックコードと一致する提案をブロックするフィルターを必ず有効にする
2. **ライセンススキャン:** Software Composition Analysis（SCA）ツールを用いて、AI生成コードのOSSライセンス適合性を検証する
3. **コピーレフトライセンスへの注意:** GPL等のコピーレフトライセンスに該当するコードパターンが混入しないよう注意する

### 第20条（AI生成コードの著作権の帰属）

1. AI生成コードの著作権の帰属については、現行法上の解釈が確立されていないことを認識する
2. 顧客への納品物にAI生成コードが含まれる場合は、顧客との契約で著作権の取り扱いを明確にする
3. 著作権に関する法的助言が必要な場合は法務部に相談する

### 第21条（IP補償（著作権補償）の活用）

1. GitHub Copilot Business / Enterprise には IP 補償（Copyright Commitment）が含まれる
2. IP 補償の適用条件（パブリックコードフィルターの有効化等）を満たしていることを確認する
3. IP 補償の適用範囲と限界を理解した上で利用する

## 第8章 セキュリティ管理

### 第22条（組織設定の必須事項）

GitHub Organization 管理者は、以下の設定を必ず実施する。

| # | 設定項目 | 設定値 |
|---|---|---|
| 1 | ライセンスプラン | Business または Enterprise |
| 2 | Suggestions matching public code | **Blocked**（パブリックコードフィルター有効） |
| 3 | Copilot を利用できるメンバー | 承認されたメンバーのみ（Selected teams/users） |
| 4 | Content Exclusion | 機密リポジトリ・ディレクトリを除外設定 |
| 5 | SSO（シングルサインオン） | 有効化（Azure AD / Okta 等と連携） |
| 6 | Secret Scanning | 有効化 |
| 7 | Audit Log | 有効化（Enterprise の場合はストリーミングも検討） |
| 8 | MCP Server 接続先 | 承認済みサーバーのみに制限 |
| 9 | **Repository Rulesets によるブランチ保護の強制** | **下記「9号」のとおり Enterprise レベルで強制適用** |

#### 第22条9号（Repository Rulesets によるブランチ保護の強制）

Enterprise 管理者は、**Enterprise レベルの Repository Rulesets** を用いて、全 Organization の全リポジトリに以下のブランチ保護ルールを強制適用する。Organization 管理者による bypass を原則として禁止し、監査可能な形で運用する。

| # | ルール項目 | 設定値 | 目的 |
|---|---|---|---|
| 1 | Require a pull request before merging | **ON（強制）** | 直接 push の禁止 |
| 2 | Required approving reviews | **1名以上（リスクに応じて2名）** | レビュー必須化 |
| 3 | Dismiss stale reviews when new commits are pushed | **ON** | 承認後の改ざん防止 |
| 4 | Require review from Code Owners | **ON** | CODEOWNERS の必須化 |
| 5 | Require approval of the most recent reviewable push | **ON** | セルフマージ防止 |
| 6 | Require conversation resolution before merging | **ON** | レビュー指摘の取りこぼし防止 |
| 7 | Require status checks to pass（CodeQL / Secret Scanning / CI） | **ON** | 品質ゲート |
| 8 | Block force pushes | **ON** | 履歴改ざん防止 |
| 9 | Require signed commits | **推奨** | なりすまし防止 |
| 10 | Restrict deletions | **ON** | 重要ブランチの削除防止 |

**リスク別のレビュー階層:**

| リスクレベル | 必要承認数 | CODEOWNERS | 追加要件 |
|---|---|---|---|
| 通常コード | 1名 | チームメンバー | CI 合格 |
| セキュリティ関連コード | 2名 | セキュリティチーム必須 | SAST 合格 |
| 本番デプロイ関連 | 2名 | SRE + Tech Lead | 変更管理チケット |
| AI生成コード（Agent Mode / Coding Agent） | 2名 | 人間レビュアー | Copilot Code Review + 静的解析 |

**CODEOWNERS の運用:**

1. 全リポジトリに `.github/CODEOWNERS` ファイルを配置することを必須とする
2. セキュリティ関連ディレクトリ（認証・暗号・インフラ等）は、セキュリティチームを CODEOWNERS に追加する
3. CODEOWNERS 不在リポジトリは四半期棚卸しで検知し、是正する

**Copilot Code Review の導入:**

1. Repository レベルで Copilot の自動レビューを有効化し、PR 作成時に一次スキャンを実行する
2. 自動レビューの結果は、人間レビュアーの判断材料として活用する
3. 自動レビューは人間レビューの代替とはせず、**第15条3項に従い人間による最終承認を必須とする**

**Bypass の管理:**

1. Enterprise Rulesets の bypass 権限は、セキュリティチームの一部メンバーに限定する
2. bypass の実施は Audit Log で記録し、月次で情報セキュリティ管理者がレビューする
3. 正当な理由のない bypass は規程違反として扱う

**段階導入計画:**

```
Phase 1（1ヶ月）  : Enterprise Rulesets で main ブランチ保護 + 1名レビュー必須
Phase 2（2-3ヶ月）: CODEOWNERS 導入 + 2名レビュー + ステータスチェック強制
Phase 3（3-6ヶ月）: Copilot Code Review 統合 + リスク別レビュー階層の適用
Phase 4（継続）   : 監査ログストリーミング + 自動違反検知の運用
```

### 第23条（アクセス管理）

1. Copilot のライセンスは、研修を修了した者にのみ付与する
2. 退職・異動時にはライセンスを速やかに回収する（SSO 連携により自動無効化を推奨）
3. ライセンスの棚卸しを四半期ごとに実施する

### 第24条（脆弱性対策）

1. AI生成コードに対して SAST（Static Application Security Testing）を実施する
2. 依存関係の脆弱性を Dependabot / Snyk 等で継続的に監視する
3. OWASP LLM Top 10 に記載されたリスク（プロンプトインジェクション等）を認識し、対策を講じる

## 第9章 プライバシー・データ保護

### 第25条（データフロー・データ処理の理解）

利用者は、Copilot のデータ処理フローを理解した上で利用する。

| データ種別 | 処理内容 | 保持期間 | 学習利用 |
|---|---|---|---|
| IDE コード補完の入力 | コード周辺のコンテキストをGitHub サーバーに送信 | リアルタイム処理後即時削除 | **されない**（Business/Enterprise） |
| Copilot Chat の入力 | プロンプトをGitHub サーバーに送信 | 最大28日間（品質改善目的） | **されない**（Business/Enterprise） |
| 提案されたコード | GitHub サーバーからIDEに返却 | 即時（キャッシュなし） | **されない**（Business/Enterprise） |
| 利用メトリクス | 利用率・提案受入率等の集計データ | 90日間 | 集計データのみ |

### 第26条（越境データ移転への対応）

1. Copilot の推論処理は米国・EU のサーバーで行われる（2026年4月時点）
2. 個人情報保護法第28条（外国にある第三者への提供）への対応として、個人データを Copilot に入力しないことで対応する（第12条参照）
3. データレジデンシー要件が厳格な業界（金融、医療等）では、プロジェクト責任者が法務部と協議の上、利用可否を判断する

### 第27条（個人情報保護法への対応）

| 対応事項 | 実施内容 |
|---|---|
| 利用目的の特定・通知 | プライバシーポリシーにAIツール利用に関する記載を追加 |
| 個人データの入力禁止 | 第12条に基づき、個人データの入力を禁止 |
| 要配慮個人情報 | Copilot への入力を全面禁止 |
| 越境移転 | 個人データを入力しないことにより対応 |

## 第10章 品質管理・人間による監視

### 第28条（人間による監視の方針）

Copilot は開発者を支援するツールであり、最終的な判断・責任は人間にある。

| 項目 | 方針 |
|---|---|
| 監視レベル | **Human-in-the-Loop**（AI の全出力を人間が確認してから採用） |
| 自動意思決定 | Copilot による自動的なコードコミット・デプロイは禁止 |
| 停止手段 | 各利用者は IDE の設定で Copilot をいつでも無効化できる |
| エスカレーション | AI生成コードに疑義がある場合はプロジェクト責任者に報告 |

### 第29条（AI 依存度の管理）

1. 開発者が AI に過度に依存し、自身のスキルが低下しないよう、定期的なスキル評価を実施する
2. AI 未使用での開発作業を定期的に行い、基本スキルの維持を図る
3. ジュニア開発者に対しては、AI 利用と基礎学習のバランスに配慮した育成計画を策定する

## 第11章 教育・研修

### 第30条（研修体系）

| 対象者 | 研修名 | 内容 | 頻度 | 修了要件 |
|---|---|---|---|---|
| 全利用者 | Copilot ガバナンス研修 | 本規程の理解、禁止事項、セキュリティ | 年1回（ライセンス付与前に必須） | テスト15点中7点以上 |
| 全利用者 | Copilot 操作研修 | IDE連携、効果的なプロンプト、活用パターン | 初回導入時 | 受講完了 |
| 管理者 | ガバナンス管理者研修 | Organization 設定、監査、インシデント対応 | 年1回 | 受講完了 |
| プロジェクト責任者 | 利用可否判断研修 | 顧客契約確認、リスク評価、判断基準 | 年1回 | 受講完了 |

### 第31条（研修テスト）

1. ガバナンス研修の理解度を確認するため、15点満点のテストを実施する
2. **7点未満の場合は不合格とし、再テストを義務付ける**
3. 合格するまで Copilot のライセンスは付与しない
4. テスト問題は AI-CoE が管理し、法規制・ガイドラインの変更に合わせて更新する

## 第12章 モニタリング・効果測定

### 第32条（モニタリング項目）

GitHub Organization 管理者は、以下の指標を定期的にモニタリングする。

| カテゴリ | 指標 | 頻度 | 閾値・基準 |
|---|---|---|---|
| **利用状況** | Copilot 週次アクティブ利用率 | 週次 | 目標: 90% |
| **利用状況** | 提案の受入率（Acceptance Rate） | 月次 | 参考値として記録 |
| **品質** | AI生成コードに対するレビュー指摘率 | 月次 | 推移を監視 |
| **セキュリティ** | セキュリティスキャン検出数（AI生成コード起因） | 月次 | 0件を目標 |
| **セキュリティ** | シークレット検出数 | 月次 | 0件を目標 |
| **インシデント** | AI生成コード起因のインシデント数 | 月次 | 0件を目標 |
| **コンプライアンス** | 研修修了率 | 四半期 | 100%必達 |
| **コンプライアンス** | ライセンス棚卸し完了率 | 四半期 | 100%必達 |

### 第33条（効果測定）

METR RCT の知見（主観と客観の約39pt乖離）を踏まえ、主観評価と客観指標の両方で効果を測定する。

| 測定方法 | 指標 | 留意事項 |
|---|---|---|
| **客観指標** | タスク完了時間、PR マージまでの時間、デプロイ頻度 | 導入前後で比較 |
| **客観指標** | コード品質（静的解析警告数、テストカバレッジ） | DORA/Cursor研究を踏まえ品質低下を監視 |
| **主観指標** | 開発者満足度アンケート | 四半期実施。客観指標と照合 |

### 第34条（報告）

AI-CoE は、モニタリング結果を四半期ごとに経営層に報告する。報告内容には、利用状況、効果測定結果、インシデント、改善提言を含める。

## 第13章 インシデント対応

### 第35条（インシデントの定義と分類）

AI生成コードに起因する以下の事象をインシデントとして取り扱う。

| レベル | 定義 | 目標対応時間 | 報告先 |
|---|---|---|---|
| **Critical** | 機密情報・個人情報の外部漏洩、本番障害 | 1時間以内 | 経営層 / 情報セキュリティ管理者 / 法務 |
| **High** | セキュリティ脆弱性の本番混入、ライセンス違反の検出 | 4時間以内 | AI責任者 / 情報セキュリティ管理者 |
| **Medium** | AI生成コードの品質問題（テスト環境での検出） | 翌営業日 | プロジェクト責任者 |
| **Low** | Copilot の誤動作・サービス障害 | 5営業日以内 | GitHub Organization 管理者 |

### 第36条（対応フロー）

```
1. 検知・報告（発見者 → プロジェクト責任者 → インシデントレベルに応じた報告先）
2. 初期対応（該当コードの修正・パッチ適用、必要に応じてサービス停止）
3. 影響範囲の特定
4. 根本原因分析（RCA）
5. 復旧・是正措置
6. 再発防止策の策定（静的解析ルールの追加、レビュープロセスの見直し等）
7. 関係者への報告（顧客への通知が必要な場合を含む）
8. 本規程・運用手順の更新（必要な場合）
```

### 第37条（規制当局への報告）

個人データの漏洩が発生した場合は、個人情報保護法に基づき、個人情報保護委員会に速やかに報告する（速報: 3〜5日以内、確報: 30日以内）。

## 第14章 規程の改廃

### 第38条（定期レビュー）

1. 本規程は **6ヶ月ごと** に見直しを行う
2. 以下のいずれかに該当する場合は、定期レビューを待たずに改訂する:
   - GitHub Copilot のサービス内容・利用規約に重大な変更があった場合
   - 関連法令・ガイドライン（個人情報保護法、AI事業者ガイドライン等）が改正された場合
   - 重大なインシデントが発生し、規程の不備が判明した場合
   - JDLA ガイドラインの新版が公開された場合

### 第39条（改訂手続き）

1. 改訂案は AI-CoE が起案し、情報セキュリティ管理者・法務部のレビューを経て、AI責任者が承認する
2. 改訂内容は全利用者に周知し、必要に応じて追加研修を実施する
